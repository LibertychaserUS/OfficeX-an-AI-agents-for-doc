# Prompt Evolution Protocol

## Purpose

This document defines how OfficeX prompt packs may evolve without becoming
opaque, unstable, or contaminated by anecdotal changes.

## Core Rule

Prompt packs are versioned assets, not invisible conversational residue.

They may improve, but they may not silently drift.

## Prompt Layers

### Constitutional Prompt Layer

High-authority cognitive framing for OfficeX.

Examples:

- core runtime cognition
- truthfulness stance
- approval stance
- tool-use stance

This layer changes rarely and requires explicit promotion.

### Role Prompt Layer

Agent-type prompts for:

- orchestrator
- constraint analyst
- writer
- asset engineer
- patch and assembly engineer
- validation and review auditor

### Domain Prompt Layer

Specialized supplements for:

- academic
- legal
- business
- compliance
- finance
- technical

### User/Project Prompt Layer

Local tone, preference, or project-specific steering.

This layer may adapt more often, but it must remain scoped.

## What May Evolve

Allowed:

- clarification structure
- safer escalation patterns
- better task decomposition hints
- stronger domain review criteria
- improved handoff formatting
- reduced ambiguity

Not allowed to drift silently:

- hard truthfulness rules
- publish rules
- approval mode semantics
- prohibited content classes
- authority precedence

## Evolution Pipeline

1. propose
2. store as draft
3. benchmark against known tasks
4. compare with current pack
5. review by human
6. promote or reject

## Proposal Sources

Prompt changes may be proposed from:

- repeated failures
- benchmark gaps
- new domain needs
- user preference patterns
- new reference learnings

Proposal sources may not directly become active packs.

## Benchmark Requirement

Before promotion, a prompt change should be tested for:

- output quality
- truthfulness stability
- escalation behavior
- handoff clarity
- prompt length control
- regression against existing strong cases

## Versioning Rule

Every promoted prompt pack should record:

- pack name
- version
- parent version
- change reason
- benchmark notes
- promotion date

## Rollback Rule

Every promoted pack must be rollbackable.

Rollback should occur when:

- false confidence increases
- truthfulness declines
- handoffs become noisier
- benchmark performance drops
- domain confusion increases

## Relationship To RAG

RAG may suggest prompt improvements.

RAG may not directly replace active prompt files.

Prompt authority must remain file-backed and versioned.
