# Development Telemetry And Hotspots

## Purpose

Development telemetry should help improve how OfficeX is built.

It should not become product authority.

## What To Observe

- frequently used harnesses
- frequently used commands
- frequently cited reference docs
- repeated runtime failure points
- repeated review findings
- repeated rejected patches
- repeated test failures

## Primary Use

Telemetry should support:

- hotspot analysis
- benchmark planning
- workflow refinement
- review-pack refinement

## Prohibited Use

Development telemetry must not automatically rewrite:

- product runtime constitutions
- active prompt packs
- authoritative memory
- approved schemas

## Promotion Rule

A hotspot may influence active development process only after:

1. observation
2. explicit note
3. review
4. approval

This keeps development optimization from turning into silent governance drift.
