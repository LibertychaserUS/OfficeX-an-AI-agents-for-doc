总结

一、OfficeX 的核心定位

OfficeX 不是普通的文档 AI，不是单纯的写作助手，也不只是 doc agents。
它更像是一个 面向高约束专业文档生产的规则驱动 Agent 系统，也可以理解成 文档界的 Claude Code / 文档优先的 Agent 系统。
它的重点不在于“通用”，甚至不想让它有太多泛化功能，而在于：
	•	专业领域的文档撰写
	•	私人订制 agents 和 memory
	•	严格根据需求自己基于流程造好轮子
	•	严格格式
	•	严格素材与引用对齐
	•	能够真正生成接近可交付物的文档

用户不只是用它“写一点东西”，而是把正式文档生产流程的一部分真的交给它。

【补充】
OfficeX 的产品核心不是“帮人写字”，而是“帮人稳定地产出高约束、强格式、强交付属性的正式文档”。

⸻

二、OfficeX 解决的问题

一句话/一段话生成文档，专业措辞和严格格式，文档纠错，能够实现素材库和文档内引用的严格对齐。
因为接入了 API，用户还可以自己造轮子，比如指定去哪个网页扒原文、扒参考文献索引等等。
但接入了 API 之后其实就不只是 doc agents 了，它实际上就是一个通用的 agents，只不过有了严格的文档操作流程和工具。

不过最好的发展点不在于本身的通用。
更大的服务是：
	•	专业领域的文档撰写
	•	私人订制 agents 和 memory
	•	严格根据需求自己基于流程造好轮子
	•	严格格式
	•	素材库和文档内引用严格对齐

【补充】
真正解决的不是“AI 能不能写”，而是“AI 能不能把高复杂正式文档做得足够稳、足够可控、足够接近交付”。

⸻

三、OfficeX 的核心卖点
	1.	规则宪法
所有问题都要被抽象出来变成规则，让 AI 遵循这些规则使用工具操作文档，就像人类使用宪法语言如 Rust 编程一样。
	2.	程序 agents + LLM agents 混合
多 agents 很多可以不用 AI，而是程序 agents 里面有内置规则。
不是所有事情都交给 AI。
规则检查、格式校验、引用对齐、路由、对象绑定、结构修复等很多都应该是程序 agents。
	3.	素材与引用系统
不只是生成文字，而是素材库、来源、引用、正文、claim 之间严格对齐。
引用和素材系统肯定要把素材库的内容抽象成为一种数据结构。
引用和素材系统可能要对整个素材库做类似 RAG 的处理。
	4.	正式文档生产秩序
它更像是文档生产的规则系统、审查系统、素材系统、对象系统，而不是普通聊天框。
	5.	可定制 workflow / 私人 memory / 私人 agents
用户可以基于自己的文档流程、素材系统、引用要求、格式要求，定制自己的 agent 和 memory。

【补充】
OfficeX 真正的差异化，是“规则驱动 + 素材对齐 + 程序化文档对象操作 + 审查闭环”。

⸻

四、关于 Word / OnlyOffice / 渲染现实

因为 Word 和 OnlyOffice 的 Word 不是完全可预测渲染器，所以最终结果不能 100% 精确执行，那至少要到 95% 以上。
还有就是不能精确预测的是纸张大小和精确的渲染位置，也就是说不精确的地方要进行分层。

这意味着：
	•	不能追求完全像素级绝对控制
	•	但必须尽可能把可控部分规则化
	•	对不可完全预测的部分要设定容错和保守策略
	•	能达到 95%+，剩余部分通过审查 agent 或人工复核收尾

【补充】
渲染问题不应该否定整个系统，而应被视为“渲染不确定性治理”问题。

⸻

五、文档问题的分层

1. 内容层 / 语义层

在内容层上，要分层。比如：
	•	参考文献的引用不能串行
	•	不能串页
	•	分页必须是一条直线，这个要清楚
	•	素材库中的素材引用要与正文严格对应
	•	对代码的引用不能乱
	•	引入的代码必须和标题一致
	•	不能出现要放后端代码的地方放到前端代码的地方了
	•	claim、source、citation、正文之间不能脱节

2. 结构层

对象应该待在它该待的位置：
	•	图和标号不能乱
	•	代码块和标题要绑定
	•	封面和正文要隔离
	•	section 层级要正确
	•	文档大纲要正确
	•	appendix、附录、图、代码块、引用对象都要在正确区域

3. 排版层

在排版层上，比如说报告的第一页，在要求的纸张大小下，正文不能跑到第一页的封面中去。
不能出现不可预测的字母间或者字间空格。
对于素材库中的素材引用，比如对代码的引用，不能出现要求以外的字体。
图片、diagram、代码、图注、表格、附录都要可读且不失真。

4. 渲染层

不能精确预测的是纸张大小和精确的渲染位置。
比如打印出来或者看电子版时太糊、比例不对、可读性的问题。
图片比例问题有一个实际图片比例和内容比例的问题，有时候可能图片比例对了但是内容比如说 diagrams 里面的一些小框比例不对，那是不行的。

【补充】
未来正式版本里，建议固定为：
语义层 / 结构层 / 排版层 / 渲染层 / 审查层。

⸻

六、失败模式与具体问题

很多问题其实是大规模文档中才会出现的问题，在小规模中不是很容易出现，但是也要注意这种问题的发生。

典型问题包括：
	•	图和标号不在同一页
	•	封面与正文串页
	•	引用和素材脱节
	•	代码标题和代码内容不一致
	•	后端代码和前端代码放错位置
	•	指定字体被替换
	•	字间距、空格异常
	•	图片模糊、比例失真、阅读体验差
	•	diagrams 内部内容比例不对
	•	大文档中连锁错位
	•	大纲识别错误
	•	page size 变化导致排版漂移
	•	Word / OnlyOffice 某些样式和对象渲染不一致

【补充】
这些不应该只被看作“bug”，而应统一进入 Failure Pattern Registry（失败模式注册表）。

⸻

七、规则“宪法”的建立方式

总之所有可能出现的问题之原因要从代码底层和实例中抽出来，被统一处理并抽象为一种底层的规则“宪法”，然后我们做 rubric 和 schema 以及工具去解决这些问题的时候就方便了。

这个宪法可以来自：
	•	自己做文档时踩的坑
	•	真实大文档中的问题
	•	Office 软件的开发日志
	•	重大 bug 修复历史
	•	论坛里人是如何解决这些问题的
	•	官方手册
	•	使用反馈平台
	•	Office 使用教学书
	•	长期人类排版、编辑、审校的经验

比如说图和标号经常容易不在同一页，就用无边框表格把它们框起来。
这里的很多具体方法只是例子，不是已经抽象成规则本体。
真正重要的是：先找出问题，再抽象为规则，再决定用什么工具和 agent 去执行。

【补充】
宪法建立流程建议是：
	1.	收集问题实例
	2.	归类到层级
	3.	提炼成规则
	4.	区分硬规则 / 软规则 / 风险规则
	5.	指定程序执行 / hybrid / LLM / 人工复核
	6.	记录来源与修复策略

⸻

八、人类在工具之前是怎么处理这些问题的

在工具之前，人主要靠：
	•	经验法
	•	模板法
	•	角色分工法
	•	反复预览与人工回查法
	•	保守策略

例如：
	•	图和图注尽量不要分开
	•	封面和正文分开
	•	长文档边写边查
	•	大图不要硬塞
	•	高风险对象宁可留白，不可串页
	•	模板优先，减少自由度
	•	编辑、排版员、校对、作者分工协作

【补充】
OfficeX 的宪法，不应该只从代码和 bug 长出来，也应该从“人类长期文档治理经验”里长出来。
也就是说，要把人类实践规则转译成机器可执行规则。

先把 OnlyOffice Word 的所有样式和功能抽出来，我们在做的时候一个一个去匹配、去完成。

【补充】

	•	结构对象
	•	高风险排版功能
	•	常见图表、代码块、附录、封面、引用对象的映射

十、图片、diagram 与视觉问题

使用的生图程序生成的图可能和实际要求的纸张大小不匹配。
这个不匹配是指在打印出来或者看电子版时，太糊、比例不对、可读性的问题。
可能要对使用的生图程序在底层上重构，让生出来的图可以拆分，并且是清晰、可以自定义拆分逻辑和展示逻辑的重构，提高文档的可读性。
这一点是基于 GU2 报告的 diagram 生成中得到的经验，也许可以从 skills 中得到灵感和可以借鉴的代码。

对于未来的 PPT 生成，必须有一个良好的素材库以及格式、样式、动画库，方便 agents 选择使用，更好的展示，还不用让 AI 自己原创，导致丑陋或者展示重心和视觉重心以及图片的展示比例不对、分辨率不对、失真。

【补充】
这里以后可以拆成两个方向：
	1.	文档内插图与 diagram 结构化生成
	2.	PPT / 演示系统的视觉资产库与样式系统

十一、未来的 PPT / 演示系统

对于未来的 PPT 生成，必须有一个良好的素材库以及格式、样式、动画库，方便 agents 选择使用，更好的展示。
还不用让 AI 自己原创导致丑陋或者展示重心和视觉重心以及图片的展示比例不对、分辨率不对、失真。

【补充】
这是 OfficeX 的长期分支，不一定是初版核心，但未来可以长成一个 OfficeX Studio。

十二、未来的文档新范式

也许未来可以发明一种格式和 office 软件，做出一个既有二维大纲（这个及以上涉及到的大纲基本上都是指 Word 里面左侧边栏的那个，应该是 Word 自动识别出来的大纲）可以正常打印，又在编写和电子展示上像无边记一样的东西。
这样就能规避很多图片表格等的展示问题。
如果无纸化的话可以采用无边记类似的，那样就不怕了，但是无边记就没有排版和大纲了，这个还得仔细讨论一下。

【补充】
这是长期愿景，不属于 OfficeX Core 初版，但很适合作为未来独立产品分支或研究方向。

十三、代码 agents、记忆、harness、skills

我在开发过程中还有一些灵感：代码 agents 以及记忆。
很多 harness 被调用了但是没有通知。
还有就是那些记忆被调用的次数多，那些 harness 被用的次数多，那些 skills 被调用的次数多，记录下来形成热点结构。
再记录调用之后经常出现的问题等等等，记录下来。
可以去做出子 skills 和子 harness，作为用户个性化的补充，或者直接修改底层。
找一找类似的算法，等等做出来一个新的索引以及记忆和 skills harness 持久化。

harness 和 skills 的区别是什么。
引用和素材系统可能要对整个素材库做类似 RAG 的处理。

【补充】
这里可以形成一个新的子系统方向：

1. 调用追踪系统

记录：
	•	哪些 skills 被调用次数多
	•	哪些 harness 被调用次数多
	•	哪些记忆被反复使用
	•	哪些组合经常一起出现
	•	调用后常见失败模式是什么

2. 热点结构

把高频调用组合、常见失败模式、常见用户行为抽象出来，形成“热点结构”。

3. 子 skills / 子 harness 自动化生成

根据热点结构做：
	•	个性化补充
	•	小型专用技能
	•	子 harness
	•	更适合用户自己的 workflow

4. 新的索引、记忆、skills/harness 持久化

这部分以后可能成为 OfficeX 甚至更大系统里的“个性化记忆与工具演化系统”。

【补充：一个当前工作定义】
	•	Skill 更像某种可复用能力、技能模块、能力包
	•	Harness 更像某种调用框架、工具封装、执行桥、能力承载器
以后这两者的边界还需要继续细化

十四、多 agents 体系

多 agents 不一定都是 AI。
很多 agent 应该是程序 agent，里面带有内置规则。
LLM agent 只处理真正需要语义理解和生成的部分。
程序 agent 负责：
	•	规则检查
	•	路由
	•	格式校验
	•	引用存在性校验
	•	样式映射
	•	对象绑定
	•	section 归位
	•	风险检测
	•	审查报告

【补充】
OfficeX 更像是一个“规则执行网络”，不是一堆会聊天的 AI 角色。

⸻

十五、数据结构与表示法

引用和素材系统肯定要把素材库的内容抽象成为一种数据结构。
图的抽象很好，但可能不只是纯图。
更好的方向可能是混合表示：
	•	文档主结构：树 / AST
	•	引用与素材关系：图
	•	宪法规则：约束系统 / 规则对象
	•	渲染控制：布局模型

【补充】
比较好的通用原则是：
	•	用树表示文档层级
	•	用图表示 source-reference-claim 关系
	•	用规则系统表示宪法和优先级
	•	用 layout model 表示渲染与分页控制

十七、产品开发原则

1. 要做好功能的分层与解耦

核心要把：
	•	编排层
	•	规则层
	•	文档对象层
	•	素材与引用层
	•	渲染与导出层

拆开。

2. 要接受重构，但要让重构便宜

底层很多概念在开发初期会很快迭代，在开发过程中经常需要重构，这是一个问题。
所以必须做好分层和解耦，不然重构会拖死整个系统。

3. MVP 应该先做“可验证内核”

在财力、token 不够的情况下，先把第一个 MVP 落地。
第一个 MVP 应该证明：
	•	在一个高复杂正式文档场景中
	•	OfficeX 比通用 AI 更稳
	•	能完成素材整合
	•	能做基本规则控制
	•	能生成接近可交付物的文档

4. 能程序化的尽量程序化

这样能节省 token，减少反复重构的成本。

二十、现阶段一句总括

OfficeX 不是一个普通的 AI 文档工具，而是一个面向高约束专业文档生产的规则驱动 Agent 系统。
它要做的是把人类长期积累的文档治理经验、Office 世界的失败模式、真实大文档中的结构问题、素材与引用的关系、以及可程序化的规则和工具统一起来，形成一种“文档宪法 + 文档对象系统 + 素材引用系统 + 多 agents 执行网络”的正式文档生产基础设施。
